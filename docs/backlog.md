## Notion integration access is per-page/per-DB, not workspace-wide

**First observed**: 2026-05-13 — My Tasks DB returned 404 to `notion_update_page_property` despite the integration being on the workspace. **Recurred**: 2026-05-14 — Disney World Trip Planner page not findable via `notion_search` for the same reason (integration not shared into that page's teamspace/section).
**Symptom**: One of these:
- `notion_search` returns empty results for a page/DB Sean confirms exists
- `notion_read` / `notion_update_*` / `notion_archive_page` returns `404 not found`
- `notion_query_database` returns `404` on a known database ID
**Root cause**: Notion's integration model is per-page/per-database, NOT workspace-wide. Even a workspace-admin-installed integration only sees pages where the integration has been explicitly added via the page's connection menu. New pages and database don't inherit access automatically.
**Fix (30 seconds)**: Open the affected page in Notion (web or app) → click `⋯` menu in the top-right → select `Connections` → search for and add the Clawdia integration. The page becomes accessible to Clawdia immediately, no Clawdia restart required.
**For teamspaces**: adding the integration to a teamspace's parent page typically grants access to descendant pages, but verify with a `notion_read` round-trip before assuming it cascades.
**Diagnostic phrasing Clawdia should use**: when she hits this, the right framing is "page exists but not shared with Clawdia integration" rather than "page doesn't exist" or "search came up empty." Currently the constructive gap-surfacing prompt rule prompts her toward the right diagnosis when the user signals content suspected to exist.
**Prevention**: when adding any new Notion DB or significant page tree that Clawdia should be able to touch (e.g. new project pages, new family-shared DBs), add the Clawdia integration via the connection menu as part of the page-creation workflow. No code change needed.

## Apple Notes bridge returns `"database disk image is malformed"`

**First observed**: 2026-05-13 13:00 UTC (morning briefing dropped the Recent Notes section silently).
**Symptom**: `journalctl -u clawdia` shows `WARNING:clawdia.briefing.sources: apple_note search HTTP 500: {"ok": false, "error": "database disk image is malformed"}`. Bridge `/notes_search` calls return the same error to every query. Briefing's per-source try/except design hides this from the user as a quietly-missing Recent Notes section.
**Root cause (likely)**: The Mac bridge reads `~/Library/Group Containers/group.com.apple.notes/NoteStore.sqlite` directly via raw SQLite. When the Notes app (or its CloudKit sync, or its WAL checkpoint) is mid-write, SQLite returns this error to external readers. The file isn't actually corrupted — the Notes app continues working fine through Apple's CoreData layer. This is a raw-SQLite-vs-app-managed-DB contention, not data loss.
**Fix (30 seconds, no data loss)**:
1. On the Mac: `osascript -e 'tell application "Notes" to quit'`
1. Wait 2 seconds
1. `open -a Notes`
1. Wait 4 seconds for it to reach steady state
1. Verify from VPS: `source /etc/clawdia/env && curl -sS -m 15 -X POST http://100.77.185.52:8733/notes_search -H "Content-Type: application/json" -H "X-Clawdia-Token: $CLAWDIA_IMESSAGE_TOKEN" -d '{"query":"a","limit":3}'` — should return real notes with titles+snippets, not the malformed error.
**Why this works**: Quitting and relaunching Notes forces macOS to checkpoint the WAL, rebuild any inconsistent indices, and reach a clean steady state. The file modification times and notes content are preserved — all changes are already committed in iCloud. The local SQLite just gets re-synthesized from the canonical CloudKit state.
**Bridge token gotcha**: The bridge auth header is `X-Clawdia-Token`, NOT `X-Auth-Token` or `Authorization`. Env var is `CLAWDIA_IMESSAGE_TOKEN` in `/etc/clawdia/env`. Wrong header = `{"error":"bad token"}`.
**When to escalate to deeper fix**: If this recurs more than ~once a week, switch the bridge to read Notes via AppleScript (`tell application "Notes"`) instead of raw SQLite. AppleScript goes through Notes' own process so it never sees the raw-file inconsistency. Estimated 30 min to rewrite the bridge's notes handlers; deferred unless frequency justifies the build.
**Last recurrence**: 2026-05-13 13:00 UTC (fix applied at 16:50 UTC, after dev-Claude session resumed in the morning; bridge healthy after).

## Audit hook firings are HYPOTHESES, not VERDICTS — dev-Claude failure pattern caught 2026-05-13

**Why this is here**: In a 24-hour window dev-Claude jumped to confident-wrong conclusions FOUR times from partial audit-log evidence (second-screenshot misread late 2026-05-12, 'yesterday vs today' timestamp confusion morning 2026-05-13, Red Wing 'fabrication' investigation midday 2026-05-13, PPE reimbursement 'she lied' near-miss afternoon 2026-05-13). The pattern is the same: dev-Claude saw partial evidence (FABRICATION_RISK warning + truncated text_preview, OR a memory.db query returning empty), formed a confident interpretation, and pushed it into conversation as fact without verifying against the right ground-truth surface.
**The fix isn't more logging — it's a procedure**: when ANY action-claim verification is in question, before saying anything to the user about it, dev-Claude MUST:
1. **Check journalctl FIRST for the tool name actually called.** `journalctl -u clawdia --since '15 min ago' --no-pager | grep AUDIT` returns the universal ground truth: what tool name was invoked, what time, with how many text chars. THIS IS STEP ONE BECAUSE IT'S SURFACE-AGNOSTIC. memory.db only proves save_memory. Notion API only proves notion_*. Drive only proves drive_*. Querying the wrong surface returns 'nothing found' and looks like fabrication when in fact the action happened on a different surface. The PPE-reimbursement near-miss (2026-05-13 19:31) is the canonical example: dev-Claude queried memory.db for a PPE entry, found nothing, almost wrote 'she lied' — but journalctl showed `notion_add_research` at the exact moment, on a different surface entirely.
1. **Read the full user-and-assistant exchange around the warning timestamp** via `SELECT ts, role, content FROM history WHERE ts BETWEEN ts-5min AND ts+5min` — not just the 250-char `text_preview` from the audit log.
1. **Check the specific data surface the tool name points to.** Only after step 1 tells you WHICH surface (memory.db / Notion / Drive / Calendar / etc.) does it make sense to query that surface for the post-action state. A memory entry that exists in the right window is proof a save_memory call happened; absence of a memory entry is NOT proof that an action didn't happen, only proof that save_memory specifically didn't happen.
1. **Verify the timestamp's date** by running `date -u` first. 'May 13 16:29:08 UTC' is not yesterday just because you read it after midnight your time. Today's date is whatever `date -u` says, not whatever you remember it being.
1. **Read the hook code if behavior is surprising** — the `prior_used_tools` flag only looks back ONE turn (or THREE turns post-2026-05-13 18:08 patch); multi-turn tool-then-summary-then-acknowledge sequences (which are common) can produce false-positive firings if the tool was outside the lookback window.
1. **TRIPWIRE: if dev-Claude is about to write strong fabrication-claim language ('she lied', 'she fabricated', 'no tool call backing it', 'this IS the fabrication failure mode'), STOP and re-verify before sending.** That word choice is the signal that verification rigor needs to be HIGHER, not lower. Four confident-wrong reads in one day all preceded by exactly that internal confidence.
1. **Only then form an interpretation and only then say something to the user about it.**
**Why this matters beyond Clawdia debugging**: The detective-control discipline applies recursively. The audit hook was built to catch Clawdia claiming actions she didn't perform. The same failure mode is available to dev-Claude when interpreting the audit hook's output. 'I read a log and concluded X' without verifying X against the underlying data is the same shape as Clawdia saying 'Saved ✅' without calling save_memory — a claim that sounds authoritative because it's adjacent to real evidence, but isn't grounded in it.
**Hook calibration item spawned by this investigation**: the `_audit_action_claims` function unions `tool_names_this_turn | tool_names_prior_turn` and checks for a match. When Clawdia does (turn N: save_memory) → (turn N+1: tool result) → (turn N+2: 'Saved ✅' summary), turn N+2's hook only sees turn N+1's tool result, not turn N's save_memory. **Fix shipped 2026-05-13 18:08 UTC**: extended lookback to 3 turns via deque-style storage. See Recently Shipped section.
**The PPE-reimbursement near-miss specifically**: at 2026-05-13 19:31 UTC, Clawdia legitimately captured an open question via `notion_add_research` THEN deleted two superseded memory entries. Her Telegram report ('PPE reimbursement is now a live Research & Backlog item — won't get lost again') was accurate. Caught it only by then running journalctl, which showed `notion_add_research` at 19:31:40 — wrong surface had been queried. The lesson is structural: querying memory.db to verify a notion_add_research call is like checking your kitchen for socks — the right answer about whether socks exist requires checking the drawer where socks live, not the room you happen to be standing in.   - Clawdia auto-creates skill: title="Correction: always check the due date before sending...", trigger="always|check|date|before", steps="1. Check the due date before sending emails", category=clawdia

### Design Rationale

**Why success_rate?** Foundation for future ranking. Skills with 1.0 (always work) are prioritized over 0.5 (experimental) in search. User feedback can tune this.
